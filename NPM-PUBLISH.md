# Публикация в npm (монорепозиторий)

Корневой `package.json` помечен как `"private": true` — **весь репозиторий одним пакетом на npm не публикуют**. Устанавливаемые зависимости — это отдельные пакеты из каталога `packages/*` (например `@openadminjs/core`, `@openadminjs/resource`, `@openadminjs/plugin-sdk`).

## Почему `examples/` и `site/` не попадут в npm сами по себе

Команда `npm publish` / `pnpm publish` запускается **из каталога конкретного пакета** (например `packages/core`). В tarball попадают **только файлы относительно этого каталога**. Соседние папки репозитория (`examples/`, `site/`, `apps/`, `prisma/` и т.д.) в архив **не включаются** — их не нужно отдельно «вырезать», если вы не публикуете корень монорепо.

Имеет смысл настроить включение файлов **внутри** каждого публикуемого пакета (см. ниже), чтобы не отдавать тесты, черновики и лишние конфиги.

## Как явно указать, что кладём в пакет (`files` — белый список)

В `package.json` публикуемого пакета добавьте поле **`files`**: в npm попадут только перечисленные пути (плюс несколько обязательных файлов вроде `package.json`, `README`, `LICENSE` — см. [документацию npm](https://docs.npmjs.com/cli/v10/configuring-npm/package-json#files)).

**Пример для пакета со сборкой в `dist/`:**

```json
{
  "name": "@openadminjs/core",
  "version": "0.1.0",
  "files": ["dist", "README.md"]
}
```

**Пример для пакета, который отдаёт исходники (`src/`):**

```json
{
  "files": ["src", "README.md"]
}
```

Так вы не отправите в реестр, например, `*.test.ts`, `vitest.config.ts`, если они лежат рядом и **не** перечислены в `files`. (Если какой-то файл всё же попадает — проверьте, не подтягивается ли он через `main` / `exports`.)

## Альтернатива: `.npmignore`

Файл **`.npmignore`** в корне **пакета** (не обязательно в корне репозитория) исключает пути из публикации. Удобен как чёрный список. Если задано и `files`, и `.npmignore`, логика сложнее: надёжнее для контроля размера пакета опираться на **`files`**.

Пример фрагмента `.npmignore` в каталоге пакета:

```
**/*.test.ts
vitest.config.ts
tsconfig.json
```

## Зависимости `workspace:*` перед публикацией

В монорепозитории зависимости между пакетами часто указаны как `"workspace:*"`. Перед публикацией их нужно заменить на **конкретные версии** опубликованных пакетов (или публиковать пакеты в правильном порядке и обновлять версии). У **pnpm** при публикации из workspace зависимости могут подставляться автоматически — см. `pnpm publish` и опцию `--no-git-checks` при необходимости.

Практический порядок:

1. Собрать то, что уходит в npm (`pnpm build` в нужных пакетах).
2. Обновить `version` в каждом публикуемом `package.json`.
3. Выполнить `pnpm publish --access public` из каталога пакета **или** через фильтр:  
   `pnpm -r publish --filter @openadminjs/core --access public`  
   (точная команда зависит от вашей политики версий и CI).

## Если бы вы публиковали весь репозиторий одним пакетом (не рекомендуется здесь)

Тогда в **корневом** `package.json` убрали бы `"private": true` и задали бы **`files`** с явным списком, **без** `examples/`, `site/`, `apps/` и т.д. Для этого проекта типичнее оставить корень приватным и публиковать только `@openadminjs/*` из `packages/`.

## Краткий чеклист

| Шаг | Действие |
|-----|----------|
| 1 | Публиковать из `packages/<имя-пакета>/`, не из корня репо. |
| 2 | Добавить `"files": [...]` в каждый публичный пакет. |
| 3 | Убедиться, что `main` / `types` / `exports` указывают на файлы, которые реально попадут в tarball. |
| 4 | Заменить `workspace:*` на версии зависимостей (или использовать поведение pnpm при publish). |
| 5 | `npm whoami`, затем `pnpm publish --access public` (для scoped-пакетов `@openadminjs/...` scope обычно публичный через `--access public`). |

После публикации установка: `npm install @openadminjs/core` или `pnpm add @openadminjs/core` — в node_modules попадёт только содержимое tarball пакета, без `examples` и `site`.
